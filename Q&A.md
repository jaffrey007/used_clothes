# 慢夏回收平台 — 答辩 Q&A

---

## 一、系统架构类

### Q：你的系统整体架构是什么？

我的系统采用**前后端分离架构**，分为三层：

```
微信小程序（用户端）  +  Vue3 后台（管理端）
              ↓
        RESTful API（FastAPI）
              ↓
          MySQL 数据库
```

- **用户端**：微信小程序，负责预约回收、查看订单、管理积分
- **管理端**：Vue3 + ElementPlus 开发的 Web 后台，负责订单审核、派单、数据统计
- **后端**：Python FastAPI，提供统一的 RESTful 接口，处理所有业务逻辑
- **数据库**：MySQL 8.0，存储用户、订单、积分等结构化数据

---

### Q：为什么选择前后端分离？

三个原因：

1. **职责清晰**：前端只管页面展示，后端只管数据处理，互不干扰，便于维护
2. **复用性强**：同一套后端 API，同时服务小程序和 Web 后台，不用写两套逻辑
3. **扩展方便**：以后要加 App 客户端，只需对接已有 API，后端不用改

---

### Q：为什么用 FastAPI + Vue + 微信小程序？（论文写的是 Flask）

- **FastAPI**（Flask 的现代化替代方案）：Flask 和 FastAPI 都是 Python 轻量级 Web 框架，设计理念相同，都用装饰器定义路由。FastAPI 性能更高，原生支持数据自动校验和接口文档，更适合实际开发。两者架构层面完全等价。
- **Vue3 + ElementPlus**：组件丰富，适合快速搭建数据管理界面，配合 ECharts 做图表方便
- **微信小程序**：目标用户是手机用户，小程序无需下载安装，微信扫码即用，门槛低，符合旧衣回收这种高频低粘性的使用场景

---

### Q：前端、后端、小程序之间如何通信？

三端统一通过 **HTTP 请求** 与后端通信，数据格式为 **JSON**。

流程：
```
前端发起 HTTP 请求（带 JWT Token）
        ↓
后端验证 Token → 处理业务逻辑
        ↓
查询或写入 MySQL
        ↓
返回 JSON 结果给前端
```

身份验证使用 **JWT（JSON Web Token）**：用户登录后后端签发 Token，前端每次请求都把 Token 放在请求头 `Authorization: Bearer <token>` 里，后端校验通过后才处理请求。

---

## 二、接口与协议类

### Q：RESTful API 是什么？

REST 是一种接口设计规范，核心思想是**用 URL 表示资源，用 HTTP 方法表示操作**。

我的系统中的例子：

| 接口 | 含义 |
|------|------|
| `GET  /api/orders` | 查询订单列表 |
| `POST /api/orders` | 创建新订单 |
| `PUT  /api/orders/5/cancel` | 取消 ID 为 5 的订单 |
| `DELETE /api/users/addresses/3` | 删除地址 |
| `GET  /api/users/me` | 获取当前登录用户信息 |

好处：接口结构清晰，前后端沟通成本低，便于扩展维护。

---

### Q：HTTP 常见请求方法有哪些？

| 方法 | 用途 | 我系统中的例子 |
|------|------|--------------|
| GET | 查询数据，不改变服务器 | 查订单、查积分、查地址列表 |
| POST | 新增数据 | 创建订单、用户登录、新增地址 |
| PUT | 修改数据 | 更新订单状态、编辑个人信息 |
| DELETE | 删除数据 | 删除收件地址 |

我的系统严格按 RESTful 规范使用了以上四种方法。

---

### Q：JWT 是什么？为什么用 JWT 而不是 Session？

JWT（JSON Web Token）是一种无状态的身份验证方式，Token 本身包含用户信息，服务端不需要存储 Session。

结构：`Header.Payload.Signature`（三段用点分隔，Base64 编码）

选择 JWT 的原因：
1. **无状态**：服务端不用存 Session，天然支持多台服务器部署
2. **适合前后端分离**：Token 由前端保存（小程序存 Storage），每次请求携带，不依赖 Cookie
3. **微信小程序没有 Cookie**，必须用 Token 方式

## 三、数据库设计类

### Q：你的数据库有哪些表？为什么这么设计？

共 **6 张表**：

```
admins          管理员表
users           用户表
addresses       用户地址表
recyclers       回收员表
orders          订单主表
order_categories 订单衣物分类明细
points_records  积分流水表
```

---

### Q：为什么订单要单独拆出 order_categories 表？

因为一个订单可以包含**多种衣物类型**（比如同时有衣服、鞋子、床品），这是一对多关系。

如果把衣物类型存在订单表里（比如用逗号分隔），会造成：
- 查询某类衣物的订单很困难
- 数据冗余，违反数据库第一范式（1NF）

所以单独建 `order_categories` 表，每种衣物类型一行，通过 `order_id` 外键关联订单。

---

### Q：订单表里为什么冗余了地址字段（addr_contact、addr_phone、addr_full）？

因为用户可能在下单后**删除地址**，如果订单只存 `address_id`，地址删了订单就看不到取件信息了。

所以下单时把地址信息**复制一份**冗余到订单表里，保证历史订单的地址信息永久可查，这是电商系统的常规做法。

---

### Q：积分为什么单独建一张 points_records 流水表？

用户表里有 `points` 字段记录当前积分余额，但光有余额不够，用户需要看**每次积分的来源和变化记录**（比如"回收奖励 +180 分"、"兑换消耗 -80 分"）。

`points_records` 表记录每一笔积分变动，字段包括：
- `change_type`：类型（earn 获得 / redeem 兑换 / adjust 手动调整）
- `amount`：变动积分（正数增加，负数扣减）
- `balance_after`：变动后余额（方便对账，不用重新计算）
- `order_id`：关联的订单（可追溯积分来源）

---

### Q：为什么用 InnoDB 引擎？为什么字符集用 utf8mb4？

- **InnoDB**：支持事务（ACID）和外键约束，适合有关联关系的业务数据。MyISAM 不支持事务，数据一致性差，不适合本系统
- **utf8mb4**：MySQL 的 `utf8` 最多存 3 字节，存不了 emoji（比如 😊），`utf8mb4` 支持完整的 4 字节 Unicode，用户昵称、备注里可能有 emoji，必须用 utf8mb4

---

### Q：你的数据库设计有哪些索引？为什么这么加？

```sql
-- users 表
INDEX idx_phone (phone)      -- 按手机号查用户
INDEX idx_status (status)    -- 筛选正常/禁用用户

-- orders 表
INDEX idx_user (user_id)     -- 查某用户的所有订单
INDEX idx_recycler (recycler_id) -- 查某回收员的订单
INDEX idx_status (status)    -- 按状态筛选订单（最高频操作）
INDEX idx_created (created_at)   -- 按时间排序/统计
```

加索引的原则：**查询频繁的字段、WHERE 条件字段、外键字段**都应该加索引，能显著提升查询速度，代价是略微增加写入时间和存储空间。

## 四、功能实现类

### Q：微信登录流程是怎么做的？

```
1. 小程序调用 wx.login() 获取临时 code
2. 把 code 发给后端 /api/auth/login
3. 后端用 code 作为 openid 标识（生产环境需调用微信服务器换取真实 openid）
4. 查找或自动创建用户记录
5. 签发 JWT Token 返回给小程序
6. 小程序保存 Token，后续请求都带上
```

这样用户每次打开小程序都能自动静默登录，不需要输入账号密码。

---

### Q：订单状态是怎么流转的？

```
待接单(0) → 已接单(1) → 回收中(2) → 已完成(3)
   ↓              ↓
 已取消(4)     已取消(4)
```

规则：
- 用户可以在"待接单"或"已接单"阶段取消订单
- 管理员派单后状态变为"已接单"
- 回收员上门后状态变为"回收中"
- 称重结算后变为"已完成"，同时给用户发放积分

---

### Q：积分是怎么计算的？

完成订单后，系统根据实际回收重量计算积分：

```
积分 = 实际重量(kg) × 单价(0.8元/kg) × 100
```

比如回收 15kg，结算 12 元，获得 1200 积分（1积分≈0.01元）。

同时更新 `users.points`（余额）和 `users.total_recycled_kg`（累计回收量），并在 `points_records` 插入一条流水记录。

## 五、安全类

### Q：密码是怎么存储的？

管理员密码用 **bcrypt** 算法加盐哈希后存储，不存明文。

bcrypt 的特点：
1. **不可逆**：无法从哈希值反推原始密码
2. **加盐**：每次哈希结果不同，防止彩虹表攻击
3. **慢哈希**：故意降低计算速度，增加暴力破解成本

登录时用 `bcrypt.checkpw(输入密码, 数据库哈希值)` 比对，不需要解密。

---

### Q：如何防止接口被未授权访问？

所有需要登录的接口都有 **JWT 验证中间件**：

1. 请求到达后端，提取 `Authorization` 头里的 Token
2. 用服务端密钥验证 Token 签名是否合法
3. 检查 Token 是否过期
4. 从 Token 中解析出用户/管理员 ID
5. 验证通过才执行业务逻辑，否则返回 401

管理员接口和普通用户接口用不同的 Token 签名方式，互相不能混用。
